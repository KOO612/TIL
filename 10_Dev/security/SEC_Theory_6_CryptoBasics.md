# SEC_Theory_6: 암호학 기초 (대칭키/비대칭키, 해시)

## 🧠 핵심 개념 한 줄 요약
암호화(대칭키/비대칭키)는 **데이터를 숨기고 나중에 복원**하는 것이 목적이고, 해시는 **데이터의 무결성(변조 여부)을 확인**하는 것이 목적이며 원천적으로 복원이 불가능하다.

## 📦 핵심 개념 상세

### 왜 EDR 개발자가 암호학을 알아야 하나
- C2 통신은 대부분 암호화되어 있어 내용을 못 보고 **패턴으로 추론**해야 함 → 암호화 개념 필요
- 파일 무결성 모니터(포트폴리오 프로젝트 2번)의 핵심 원리가 **해시**

### 1️⃣ 대칭키 암호화 (Symmetric Encryption)
- 비유: 집 열쇠 하나로 문을 잠그고 여는 것 — 잠근 사람과 여는 사람이 같은 열쇠를 씀
- 암호화(잠그기)와 복호화(풀기)에 **동일한 키** 사용
- 장점: 속도 빠름 (연산 단순)
- 단점: 그 "같은 열쇠"를 상대방에게 안전하게 전달하는 게 문제
- 대표 알고리즘: **AES**

```
평문 "hello" --[AES + 키A]--> 암호문 "x8f2..."
암호문 "x8f2..." --[AES + 키A(동일)]--> 평문 "hello"
```

> 랜섬웨어가 파일을 암호화할 때도 대부분 AES를 씀 — 속도가 빨라야 짧은 시간에 대량의 파일을 암호화할 수 있기 때문.

### 2️⃣ 비대칭키 암호화 (Asymmetric Encryption / Public Key)
- 비유: 우체통 — 누구나 편지를 넣을 수 있지만(공개키), 우체통을 열어볼 수 있는 건 열쇠를 가진 집주인뿐(개인키)
- **한 쌍**의 키(공개키/개인키) 사용. 공개키로 암호화한 건 오직 짝이 되는 개인키로만 복호화 가능
- 공개키(Public Key): 누구에게나 공개 가능 (우체통 투입구)
- 개인키(Private Key): 절대 공개 금지, 본인만 보관 (집 안의 열쇠)
- 장점: 대칭키의 "같은 열쇠 전달" 문제 해결 — 공개키는 유출돼도 상관없음
- 단점: 연산 복잡해서 대칭키보다 느림
- 대표 알고리즘: RSA, ECC (Elliptic Curve Cryptography)

```
누구나 → [공개키로 암호화] → 암호문 → [개인키로만 복호화 가능] → 서버(집주인)만 열람
```

### 하이브리드 방식 (실전에서는 둘을 섞어 씀)
HTTPS가 대표적:
1. 처음 연결 시 **비대칭키**로 안전하게 "대칭키를 교환" (느리지만 안전하게 전달)
2. 그 이후 실제 데이터는 **대칭키**로 빠르게 암호화/복호화 (속도 확보)

> C2 통신도 대부분 TLS/HTTPS 방식을 그대로 흉내 내어 정상 트래픽처럼 위장함 → EDR/네트워크 보안이 "내용은 못 보고 패턴만 본다"는 이유가 여기서 명확해짐

### 3️⃣ 해시 (Hash)
- 비유: 지문 — 사람마다 고유하고, 지문만 보고 원래 사람 얼굴을 역으로 알아낼 수 없음
- 임의 길이의 데이터를 **고정 길이의 값**으로 변환하는 단방향 함수
- 핵심 특징 3가지:
  1. **단방향성**: 해시값에서 원본 데이터를 역산 불가 (암호화와의 결정적 차이 — 해시는 "복호화" 개념 자체가 없음)
  2. **고정 길이**: 원본이 1MB든 1GB든 결과는 항상 같은 길이 (SHA-256은 항상 256비트=64자리 16진수)
  3. **눈사태 효과 (Avalanche Effect)**: 원본이 단 1비트만 바뀌어도 해시값이 완전히 다르게 나옴
- 대표 알고리즘: SHA-256, MD5(현재 취약해서 보안 용도로는 사용 안 함)

```
SHA256("hello world")      → "b94d27b9934d3e08a52e52d7da7dacc..."
SHA256("hello world!")     → "7509e5bda0c762d2bac7f90d758b5b22..." (1글자만 추가해도 완전히 다른 값)
```

## 🔗 EDR과의 연관성
포트폴리오 프로젝트 2번(파일 무결성 모니터)의 핵심 원리가 해시.

- **파일 무결성 검증**: 파일의 SHA-256 해시를 미리 계산해두고, 나중에 다시 계산해서 값이 다르면 "파일이 변경됐다"고 판단 — 원본 내용을 저장할 필요 없이 짧은 해시값만 비교
- **악성코드 시그니처 매칭**: 알려진 악성코드 파일의 해시값을 DB로 관리 (VirusTotal이 이 방식) — 파일 해시가 일치하면 즉시 "알려진 악성코드"로 판별
- **랜섬웨어 탐지**: 파일이 암호화되면 해시값이 완전히 달라짐 → 짧은 시간에 대량의 파일 해시가 동시에 바뀌면 랜섬웨어 행위로 의심

### 암호화 vs 해시 핵심 차이

| 항목 | 암호화 (Encryption) | 해시 (Hash) |
|---|---|---|
| 목적 | 데이터를 숨기고, 나중에 복원 | 데이터의 무결성 확인 (변조 여부) |
| 복원 가능? | O (키로 복호화 가능) | X (원천적으로 불가능, 단방향) |
| 결과 길이 | 원본에 따라 가변적 | 항상 고정 길이 |
| EDR 활용 | C2 통신 분석, VPN | 파일 무결성 모니터링, 악성코드 시그니처 |

## ⚠️ 자주 하는 오해/실수
- 해시를 "암호화의 일종"으로 오해하기 쉬운데, 해시는 애초에 복호화라는 개념이 없는 별도 카테고리
- MD5를 여전히 보안 용도(무결성 검증 등)로 써도 된다고 오해하기 쉬운데, 현재는 충돌 공격에 취약해 보안 목적으로는 SHA-256 이상을 사용해야 함
- 비대칭키가 항상 더 안전하다고 생각하기 쉬운데, 실제로는 속도 문제 때문에 대칭키와 함께 하이브리드로 쓰는 것이 표준 (HTTPS/TLS 방식)
