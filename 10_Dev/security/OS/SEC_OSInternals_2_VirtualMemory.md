# SEC_OSInternals_2_VirtualMemory

## 🧠 핵심 개념 한 줄 요약
각 프로세스는 자신만의 독립된 가상 주소 공간을 가진 것처럼 착각하도록 OS(페이지 테이블 + MMU)가 관리해주며, 이 구조를 알아야 프로세스 인젝션과 메모리 스캐닝 기반 탐지를 이해할 수 있다.

## 📦 핵심 개념 상세

### 비유: 가상 메모리 = 아파트 단지의 가짜 주소 체계
- **가상 주소(Virtual Address)**: 아파트 단지의 "동/호수 표시판". 프로세스 A와 B가 같은 "101동 101호"를 가리켜도 실제로는 완전히 다른 물리적 공간일 수 있음.
- **물리 주소(Physical Address)**: 실제 RAM 칩 위의 진짜 위치.
- **페이지 테이블(Page Table)**: "동/호수 → 실제 GPS 좌표" 변환표. CPU의 MMU(Memory Management Unit)가 이 표를 참고해 가상 주소를 물리 주소로 매번 변환.
- **페이징(Paging)**: 메모리를 고정 크기 블록(보통 4KB, "페이지")으로 잘라 관리하는 방식.

### 왜 이렇게 복잡하게 만드는가
1. **격리(Isolation)**: 프로세스 A가 프로세스 B의 실제 메모리를 직접 건드릴 수 없음 → 보안의 기본 전제.
2. **효율**: 물리 메모리 부족 시 안 쓰는 페이지를 디스크(페이지 파일)로 밀어냄(swap out).

### Windows 프로세스 주소 공간 구조 (64비트, 단순화)
```
높은 주소 ─────────────────────────
          커널 공간 (Kernel Space)
          ← 모든 프로세스가 같은 가상 주소로 "공유"하지만 유저 모드 접근 불가
─────────────────────────────────── (유저/커널 경계선)
          스택 (Stack) - 지역변수, 함수 호출
          힙 (Heap) - new/malloc 할당 영역
          로드된 DLL들 (kernel32.dll, ntdll.dll 등)
          실행 파일 이미지 (.exe 매핑 영역)
낮은 주소 ─────────────────────────
```
커널 공간은 모든 프로세스에서 동일한 가상 주소를 가리키지만, 유저 모드(Ring 3) 코드가 접근을 시도하면 즉시 Access Violation 예외 발생.

## 🔗 EDR과의 연관성
- **프로세스 인젝션 탐지**: 정상 코드는 실행 파일 이미지나 로드된 DLL 영역에서 실행됨. 인젝션된 코드는 보통 힙이나 `VirtualAllocEx()`로 별도 할당한 영역에서 실행되는데, 이런 영역은 원래 실행 권한이 없어야 정상. EDR은 "실행 권한이 있는데 파일에 매핑되지 않은 메모리 영역"을 `VirtualQueryEx()` 등으로 조사해 탐지.
- **Fileless Malware(파일 없는 악성코드) 탐지**: 디스크가 아니라 메모리 안에서만 복호화·실행되는 악성코드는 가상 주소 공간을 직접 스캔해야 잡을 수 있음.
- **RWX 페이지 탐지**: 쓰기(Write)+실행(Execute) 권한이 동시에 켜진 메모리 페이지는 정상 프로그램에서는 드물어서, 그 자체가 강력한 EDR 탐지 신호가 됨.

## ⚠️ 자주 하는 오해/실수
- "가상 주소 = 실제 RAM 위치" → 틀림. 페이지 테이블을 거쳐야만 물리 주소로 변환됨.
- "두 프로세스가 같은 가상 주소값을 가지면 같은 데이터를 가리킨다" → 틀림. 프로세스별로 독립된 페이지 테이블이 있어 같은 가상 주소값이어도 실제 물리 메모리는 전혀 다를 수 있음.
- "커널 공간도 프로세스마다 다른 가상 주소를 쓴다" → 틀림. 커널 공간은 모든 프로세스에서 동일한 가상 주소를 공유하지만 유저 모드에서는 접근 불가.
