# SEC_OSInternals_1_ProcessThreadStructure

## 🧠 핵심 개념 한 줄 요약
프로세스는 자원을 소유하는 "회사"이고 스레드는 실제 일을 하는 "직원"이다. PEB(프로세스 전체 정보)와 핸들 테이블(자원 목록)은 프로세스당 하나, TEB(스레드별 개인 정보)는 스레드마다 하나씩 존재한다.

## 📦 핵심 개념 상세

### 비유: 프로세스 = 회사 하나
- **프로세스(Process)**: 회사 전체. 자원(메모리, 파일 핸들, 권한)을 소유하는 단위.
- **스레드(Thread)**: 회사에서 일하는 직원. 실제로 코드를 실행하는 주체. 프로세스 하나에 스레드가 여러 개 있을 수 있음.
- **핸들 테이블(Handle Table)**: 회사의 "출입증/열쇠 보관함". 파일, 레지스트리 키, 뮤텍스 등 열어놓은 자원마다 번호표(핸들)를 발급받고 관리. **프로세스당 하나**이며 소속 스레드들이 공유.
- **PEB (Process Environment Block)**: 회사의 "총무팀 서류철". 로드된 DLL 목록, 커맨드라인, 환경변수 등 프로세스 전체 정보. **프로세스당 하나**, 유저 모드에서 접근 가능.
- **TEB (Thread Environment Block)**: 각 직원의 "개인 사물함". 스레드 로컬 스토리지, 예외 처리 체인 등 스레드 고유 정보. **스레드마다 하나씩** 존재.

### 핵심 구분 포인트
| 구조체 | 존재 개수 | 비유 |
|---|---|---|
| PEB | 프로세스당 1개 | 총무팀 서류철 |
| 핸들 테이블 | 프로세스당 1개 (스레드들이 공유) | 회사 전체 열쇠 보관함 |
| TEB | 스레드마다 1개 | 개인 사물함 |

> 헷갈리기 쉬운 부분: "핸들 테이블도 스레드마다 있을 것 같다"는 오해가 흔한데, 실제로는 프로세스 소유이고 모든 스레드가 같은 테이블을 공유한다. 개별 스레드 단위 정보는 TEB에만 있다.

## 🔗 EDR과의 연관성
- **PEB**: 악성코드가 디버거 탐지 회피 시 PEB의 `BeingDebugged` 플래그를 확인하는 경우가 많음 → EDR은 반대로 이 필드 접근/조작을 감시해 안티 디버깅 시도를 탐지.
- **핸들 테이블**: EDR이 "이 프로세스가 어떤 파일/레지스트리에 접근했는가"를 추적할 때, 핸들 생성/종료 시점을 후킹하는 것이 기본 전략.
- **PEB의 로드된 모듈 목록**: 프로세스 인젝션(DLL 인젝션, T1055) 탐지 시 "예상치 못한 DLL이 로드됐는가" 확인의 핵심 데이터.
- 핸들 테이블(프로세스 단위)과 TEB(스레드 단위)를 구분해야, 인젝션 탐지 로직에서 "어떤 스레드가 이 핸들 조작을 했는지"까지 추적할 수 있음.

## ⚠️ 자주 하는 오해/실수
- 핸들 테이블이 스레드마다 별도로 존재한다고 착각 → 실제로는 프로세스당 하나, 스레드들이 공유.
- PEB와 TEB를 혼동 → PEB는 프로세스 전체 정보(DLL 목록 등), TEB는 스레드 개별 정보(TLS 등).
- 디버거 탐지가 TEB에서 이뤄진다고 착각 → `BeingDebugged` 플래그는 PEB에 있음.
