# SEC_OSInternals_3_KernelUserMode

## 🧠 핵심 개념 한 줄 요약
CPU는 실행 권한을 Ring 0(커널 모드, 모든 것 가능)과 Ring 3(유저 모드, 제한적)로 하드웨어 레벨에서 구분하며, 유저 모드는 시스템 콜을 통해서만 커널에 요청을 전달할 수 있다 — 이 경계가 EDR을 유저 모드로 만들지 커널 드라이버로 만들지를 가르는 기준이다.

## 📦 핵심 개념 상세

### 비유: Ring = 건물의 출입 등급
- **Ring 0 (커널 모드)**: 건물의 "중앙 관제실 + 전기실 + 보안실". CPU·메모리·하드웨어에 직접 명령 가능. 실수하면 시스템 전체가 블루스크린.
- **Ring 3 (유저 모드)**: 건물의 "일반 사무실 층". 자기 프로세스 안에서만 자유롭게 활동. 전기실(하드웨어)을 만지려면 반드시 관제실(커널)에 요청 — 이것이 **시스템 콜(System Call)**.

```
[유저 모드 프로그램 (Ring 3)]
        │  "파일 좀 읽어줘" 요청
        ▼
   시스템 콜 (System Call) — 이 문을 통해서만 넘어갈 수 있음
        │
        ▼
[커널 모드 (Ring 0)] ─ 실제로 하드웨어(디스크)에 명령
        │
        ▼
   결과를 다시 유저 모드로 반환
```

### 왜 이렇게 나눴는가
- **안정성**: 일반 프로그램이 실수로 하드웨어를 잘못 건드리면 시스템 전체가 정전되므로, 반드시 커널을 거치게 함.
- **보안**: 악성코드가 Ring 3에만 머물면 자기 프로세스 안에서만 활동 가능. Ring 0을 장악하면 시스템 전체, EDR 자체까지 무력화 가능.

### EDR 개발과의 모드별 비교
| 모드 | 예시 | 특징 |
|---|---|---|
| 유저 모드 EDR 에이전트 | 대부분의 프로세스/파일 감시 로직 | 개발·디버깅 쉬움, 크래시 나도 시스템은 안전, 커널 레벨 은닉(루트킷)엔 취약 |
| 커널 모드 드라이버 | Minifilter, `PsSetCreateProcessNotifyRoutine` | 우회 어려운 근본적 감시, 실수 시 블루스크린, 드라이버 서명 필요 |

- 로드맵 2단계(유저 모드 EDR)는 Ring 3 작업, 3단계(커널 드라이버)부터 Ring 0.
- 루트킷이 위험한 이유: Ring 0을 장악하면 유저 모드 EDR이 보는 정보 자체를 조작해 프로세스를 숨길 수 있음 → 강력한 EDR은 결국 커널 모드까지 내려가야 함.

## 🔗 EDR과의 연관성
- 유저 모드 vs 커널 모드 선택이 EDR 제품의 강력함과 위험성 트레이드오프를 결정.
- 루트킷 탐지는 결국 "유저 모드에서 보이는 정보"와 "커널 모드에서 보이는 진짜 정보"를 교차 검증하는 방식으로 이뤄짐.
- 3단계(WDK, 커널 콜백, 미니필터)로 넘어가기 전 반드시 이 경계 개념이 잡혀 있어야 함.

## ⚠️ 자주 하는 오해/실수
- "유저 모드 프로그램도 마음만 먹으면 하드웨어에 직접 접근 가능하다" → 틀림. CPU가 하드웨어 명령어를 Ring 3에서 실행 시 예외를 발생시켜 강제 차단 — 소프트웨어 제약이 아니라 하드웨어 레벨 제약.
- "시스템 콜은 일반 함수 호출과 같다" → 틀림. Ring 3 → Ring 0 전환이라는 특수 CPU 명령(`syscall`, `int 0x2e` 등)을 거치며 컨텍스트 스위칭 비용이 발생.
