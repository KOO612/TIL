# SEC_EDR_6 — 커널 레벨 EDR 개념 (커널모드 vs 유저모드, 커널 콜백, 미니필터)

## 🧠 핵심 개념 한 줄 요약
유저모드 EDR은 악성코드와 "동급 권한"이라 공격받을 수 있지만, **커널모드(Ring 0)**에서 동작하는 EDR은 유저모드 악성코드가 직접 방해할 수 없는 위치에서 감시·강제 차단이 가능하다.

---

## 📦 핵심 개념 상세

### 1. 커널모드 vs 유저모드 — Ring 구조
```
Ring 0 (커널모드) ← 가장 높은 권한. 하드웨어 직접 제어, 메모리 전체 접근
Ring 3 (유저모드) ← 일반 프로그램 동작. 제한된 권한
```
- 비유: Ring 3(유저모드) = 아파트 개별 세대(자기 집 안에서만 자유), Ring 0(커널모드) = 관리사무소(마스터키로 전체 통제).
- 지금까지 배운 ETW 구독, API 후킹, 프로세스/DLL/레지스트리 감시는 전부 **Ring 3**에서 동작 → "같은 층에 있는 감시자"라 악성코드가 직접 공격 가능.
- 유저모드 프로세스가 커널 자원에 접근하려면 반드시 **syscall**을 통해 커널의 검증/허가를 받아야 함 → 커널모드 코드는 유저모드가 강제로 건드릴 수 없는 위치.

### 2. 커널 드라이버 (.sys)
- 일반 `.exe` = "세입자", `.sys` 커널 드라이버 = "관리사무소 직원 자격증을 가진 특수 프로그램".
- Windows 최신 버전은 **디지털 서명 검증을 통과하지 못한 드라이버는 로드 자체를 차단** (Driver Signature Enforcement).

**EDR의 일반적 구조**
```
[유저모드]  EDR 에이전트 프로세스 (UI, 정책 관리, 클라우드 통신, 로그 전송)
                       ↕ (통신)
[커널모드]  EDR 커널 드라이버 (.sys) ← 실제 감시/차단은 대부분 여기서 수행
```

### 3. 커널 콜백 (Kernel Callback)
Windows 커널이 특정 이벤트 발생 시 **등록된 함수를 자동 호출**해주는 메커니즘.

| API | 감지하는 이벤트 |
|---|---|
| `PsSetCreateProcessNotifyRoutineEx` | 프로세스 생성/종료 |
| `PsSetCreateThreadNotifyRoutine` | 스레드 생성/종료 |
| `PsSetLoadImageNotifyRoutine` | DLL/실행파일 메모리 로드 시 |
| `CmRegisterCallback` | 레지스트리 키 값 변경 |
| `ObRegisterCallbacks` | 프로세스/스레드 핸들 획득 시도 (자기보호의 핵심) |

**2단계(유저모드 스냅샷) vs 3단계(커널 콜백) 비교**

| 항목 | 유저모드 프로세스 스냅샷 | 커널 콜백 |
|---|---|---|
| 실행 위치 | Ring 3 | Ring 0 |
| 감지 방식 | 폴링 (놓칠 수 있음) | 이벤트 발생 즉시 자동 호출 (100% 감지) |
| 우회 가능성 | 악성코드가 감시 회피/EDR 프로세스 공격 가능 | 유저모드 악성코드가 직접 방해 불가 |
| 차단 가능 여부 | 감지만 가능, 차단은 별도 구현 | 생성 자체를 거부(차단) 가능 |

**핵심**: `ObRegisterCallbacks`는 핸들 권한을 실시간으로 수정하거나 접근을 거부할 수 있어서, 악성코드가 EDR 프로세스를 `OpenProcess()`로 열어 종료시키려 할 때 **"종료 권한만 쏙 빼고" 핸들을 내어주는 자기보호(Self-Protection)**가 가능함.

### 4. 미니필터 드라이버 (Minifilter Driver)
- 파일 시스템 접근(열기/읽기/쓰기/삭제)에 특화된 정교한 감시+차단 프레임워크 (Microsoft의 Filter Manager 구조 사용).

```
사용자가 파일을 연다 (CreateFile 호출)
        ↓
[Filter Manager]가 등록된 모든 미니필터에게 순서대로 알림
        ↓
   미니필터 A (안티바이러스) → 검사 → 허용/차단
        ↓
   미니필터 B (클라우드 동기화 등) → 자기 처리
        ↓
   실제 파일 시스템(NTFS)
```
- **핵심 특징**: 파일이 실제로 변경되기 **전에** 개입 가능 → 랜섬웨어의 암호화 쓰기 요청을 사전에 차단 (예: Controlled Folder Access).

**커널 콜백 vs 미니필터 비교**

| 항목 | 커널 콜백 (Ps*, Cm*, Ob*) | 미니필터 드라이버 |
|---|---|---|
| 대상 | 프로세스/스레드/레지스트리/핸들 | 파일 시스템 I/O |
| 구현 복잡도 | 상대적으로 단순 | 훨씬 복잡 (별도 프레임워크, 우선순위 관리) |
| 개입 시점 | 이벤트 발생 통지 (일부 차단 가능) | I/O 요청 단계에서 사전/사후 개입 |
| 대표 활용 | 프로세스 생성 차단, 자기보호 | 랜섬웨어 실시간 차단 |

## 🔗 전체 그림 — EDR 아키텍처 계층

```
[유저모드 - Ring 3]
  ├─ ETW 구독            — OS가 알려주는 이벤트 수신
  ├─ API 후킹            — 프로세스 내부 함수 호출 가로채기
  └─ 프로세스/DLL/레지스트리 감시 — 폴링 + 이벤트 기반 감시
              ↕ (한계: 악성코드와 동급 권한이라 우회/공격 가능)

[커널모드 - Ring 0]
  ├─ 커널 콜백 (Ps*, Cm*, Ob*)  — 프로세스/스레드/레지스트리/핸들 감시+차단
  └─ 미니필터 드라이버          — 파일 시스템 I/O 실시간 검사+차단
              ↕ (장점: 유저모드 악성코드가 직접 방해 불가, 강제 차단 가능)
```

실전 EDR은 두 계층 모두 사용: 유저모드는 "빠른 1차 필터 + 클라우드 통신/정책 관리", 커널모드는 "신뢰할 수 있는 최종 판단 + 강제 차단 + 자기보호".

---

## ⚠️ 자주 하는 오해/실수
1. "커널 드라이버만 있으면 유저모드 감시는 필요 없다" → **아님**. 커널 코드 버그는 블루스크린(BSOD)으로 이어지므로, 가벼운 검사는 유저모드, 중요한 차단/보호만 커널모드로 역할 분담.
2. "커널 드라이버는 아무나 만들어서 배포할 수 있다" → **아님**. Windows 10 이후 서명 정책이 매우 엄격함. (2024년 CrowdStrike 대규모 장애는 서명된 정상 드라이버 안의 로직 결함 — "서명됨 = 무조건 안전"은 아니라는 교훈.)
3. "미니필터와 커널 콜백은 같은 것이다" → **아님**. 커널 콜백은 프로세스/스레드/레지스트리/핸들 등 범용 이벤트, 미니필터는 파일 시스템 I/O 전용 별도 프레임워크.
