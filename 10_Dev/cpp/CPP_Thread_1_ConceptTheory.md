## 🧠 핵심 개념 한 줄 요약

> **스레드 = 프로세스 안에서 실행 흐름을 나눈 단위 — 메모리는 공유하지만 스택은 개별 소유**

---

## 📦 프로세스 vs 스레드

```
프로세스 (건물 = 독립된 메모리 공간)
├── 스레드 1 (직원 1) — 자기만의 스택
├── 스레드 2 (직원 2) — 자기만의 스택
└── 스레드 3 (직원 3) — 자기만의 스택
        ↑
        └── 전역 변수, 힙 메모리는 모두 공유
```

| 구분 | 프로세스 | 스레드 |
|------|----------|--------|
| 메모리 공간 | 독립적 (자기만의 주소 공간) | 같은 프로세스 내에서 공유 |
| 생성 비용 | 무거움 | 가벼움 |
| 개별 소유 자원 | 전부 | 스택, 레지스터, 실행 위치(PC)만 |
| 공유 자원 | 없음 | 전역 변수, 힙, 파일 핸들 등 |

> **핵심**: "메모리를 공유한다"는 사실 자체가 동기화 문제(데이터 경합)의 근본 원인

## ⚡ 동시성(Concurrency) vs 병렬성(Parallelism)

| 구분 | 의미 | 조건 |
|------|------|------|
| 병렬성 | 여러 스레드가 물리적으로 실제 동시 실행 | CPU 코어가 여러 개 필요 |
| 동시성 | 하나씩 번갈아 실행하지만 매우 빨라서 동시처럼 보임 | 코어 1개여도 가능 |

```
시간 →
CPU: [스레드A 실행][스레드B 실행][스레드A 실행][스레드C 실행]...
```

- 이 전환 작업을 **컨텍스트 스위칭(Context Switching)**이라 함
- OS가 스레드 A의 실행 지점을 저장 → 스레드 B로 CPU 전환 → 나중에 A로 복귀 시 저장된 지점부터 재개

> ⚠️ 스레드를 코어 수보다 지나치게 많이 만들면 컨텍스트 스위칭 오버헤드로 오히려 성능 저하 가능

---

## 🎯 스레드가 필요한 두 가지 목적

| 목적 | 설명 | 코어 개수 영향 |
|------|------|----------------|
| 응답성(Responsiveness) | UI가 멈추지 않도록 무거운 작업을 워커 스레드로 분리 | 코어 1개여도 의미 있음 |
| 처리량(Throughput) | 큰 작업을 나눠 병렬 처리해 전체 완료 시간 단축 | 코어가 여러 개여야 실질 효과 |

> 데스크탑 앱 개발에서 가장 흔한 이유는 **응답성** — "저장" 같은 무거운 작업을 메인 스레드에서 실행하면 "응답 없음" 상태 발생

---

## 🔄 스레드 생명주기 (상태)

```
[생성됨] → [Ready] ⇄ [Running] → [Terminated]
                          ↓
                     [Blocked]
              (뮤텍스 락 대기, I/O 대기, Sleep 등)
```

| 상태 | 의미 |
|------|------|
| Ready | 실행 준비 완료, CPU 배정 대기 중 |
| Running | 실제로 CPU에서 명령어 실행 중 |
| Blocked | 무언가를 기다리는 중 — CPU 전혀 사용 안 함 |

- OS **스케줄러**가 우선순위/타임 슬라이스 등을 고려해 어떤 스레드를 Running으로 바꿀지 결정
- `WaitForSingleObject`/`t.join()` 호출 시 해당 스레드는 대상이 끝날 때까지 Blocked 상태로 진입 (CPU 낭비 없이 대기)

## ⚠️ 데이터 경합 (Race Condition) — 발생 원리

`g_counter++`는 겉보기엔 한 줄이지만 실제로는 3단계로 분리됨:

```
1. 읽기: g_counter 값을 레지스터로 복사
2. 계산: 레지스터 값 + 1
3. 쓰기: 레지스터 값을 g_counter에 다시 저장
```

```
초기값: g_counter = 0

스레드 A: 1. 읽기 (0)
스레드 B: 1. 읽기 (0)         ← 컨텍스트 스위칭으로 끼어듦
스레드 A: 2. 계산 (0+1=1)
스레드 A: 3. 쓰기 (1)
스레드 B: 2. 계산 (0+1=1)      ← B는 옛날 값(0) 기준으로 계산
스레드 B: 3. 쓰기 (1)          ← A의 결과를 덮어씀

결과: 1 (기대값 2가 나와야 하는데 틀어짐)
```

> 타이밍에 따라 결과가 달라지므로 **재현이 어려운 버그**의 대표 사례

## 🛡️ 동기화 도구 비교

| 도구 | 원리 | 사용 상황 |
|------|------|----------|
| `std::mutex` + `lock_guard` | "한 번에 한 스레드만 들어올 수 있는 문" | 여러 변수를 묶은 복합 로직 보호 |
| `std::atomic` | CPU 하드웨어 차원의 원자적(읽기-계산-쓰기 한 덩어리) 명령어 | 단순 변수(카운터, 플래그) 증감 |
| `std::condition_variable` | 조건이 충족될 때까지 대기 | 생산자-소비자 패턴 |

```cpp
std::mutex g_mutex;
int g_counter = 0;

void increment() {
    std::lock_guard<std::mutex> lock(g_mutex); // RAII로 자동 잠금/해제
    g_counter++;
} // 소멸자에서 자동 unlock() — 8강 RAII 개념과 직결
```

> `lock_guard`를 쓰는 이유: `lock()`/`unlock()`을 수동으로 호출하면 중간 예외 발생 시 `unlock()` 누락 → 데드락 위험. RAII로 자동 보장

## 🔒 데드락 (Deadlock)

```
스레드 A: 자물쇠1 잠금 → 자물쇠2 대기
스레드 B: 자물쇠2 잠금 → 자물쇠1 대기
→ 서로 영원히 기다리며 멈춤
```

> **예방 원칙**: 여러 잠금을 걸 때는 항상 동일한 순서로 잠글 것

## ⚖️ Orthodox C++ vs Modern C++

| 항목 | Orthodox (Win32) | Modern C++ (C++11~) |
|------|-------------------|----------------------|
| 스레드 생성 | `CreateThread()` + 함수 포인터 | `std::thread` + 함수/람다/멤버함수 |
| 스레드 대기 | `WaitForSingleObject(hThread, INFINITE)` | `t.join()` |
| 동기화 | `CRITICAL_SECTION`, Win32 `Mutex` | `std::mutex`, `std::atomic` |
| 플랫폼 | Windows 전용 | 크로스플랫폼 (Windows/macOS/Linux 동일 코드) |

---

## 🏭 MFC/Win32 실무 연관

- MFC는 `AfxBeginThread()`로 Win32 `CreateThread`를 감싸서 제공
- **UI 요소(HWND, 컨트롤)는 반드시 자신을 생성한 스레드(메인 UI 스레드)에서만 접근** — 워커 스레드에서 직접 UI 수정 시 크래시 위험
- 워커 스레드 → UI 스레드 통신은 `PostMessage`로 위임

---

## 🌐 크로스플랫폼 주의사항

```cpp
// ✅ 표준 라이브러리 — 플랫폼 무관하게 동일 코드
#include <thread>
#include <mutex>
#include <atomic>

// ❌ Windows 전용 — macOS/Linux에서 사용 불가
// CreateThread, CRITICAL_SECTION, WaitForSingleObject
```

- 회사(Win32 스레드)와 집(std::thread) 코드는 문법이 다르지만 개념(생성/대기/동기화)은 동일
- ImGui 등 크로스플랫폼 전환 시 스레드 코드는 `std::thread` 계열로 통일 권장

## ⚠️ 자주 하는 실수

| 실수 | 올바른 방법 |
|------|------------|
| 스레드 수를 코어 수보다 과도하게 많이 생성 | 컨텍스트 스위칭 오버헤드 고려해 적정 수 유지 |
| `g_counter++` 같은 연산을 원자적이라고 착각 | 실제론 읽기/계산/쓰기 3단계 — mutex/atomic으로 보호 |
| `mutex.lock()`/`unlock()` 수동 호출 | `lock_guard`로 RAII 기반 자동 해제 사용 |
| 여러 잠금을 서로 다른 순서로 잠금 | 항상 동일한 순서로 잠가 데드락 예방 |
| 워커 스레드에서 UI 컨트롤 직접 수정 | `PostMessage`로 메인 UI 스레드에 위임 |
| `std::thread` 소멸 전 `join()`/`detach()` 누락 | 둘 중 하나 반드시 호출 (안 하면 `std::terminate()`) |
